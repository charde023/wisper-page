# 맥(M4 Max) 로컬 TechBridge-KR 전사→학습노트 파이프라인 구현 기획서

> 대상: `wisper-page/workflow/youtube` (TechBridge-KR 유튜브 → Whisper 전사 → Claude 한국어 학습노트 → Obsidian)
> 목적: **윈도우+CUDA(시그마) 전용 전사 인프라를 맥(스미스 M4 Max)에서 100% 로컬로 재현**하고, 더 고도화한다.

---

## 0. 결론 (TL;DR)

- **이식 대상은 사실상 딱 하나 = 전사 엔진.** 수집·큐레이션·프로비넌스·인덱스 스크립트(`*.py`)는 이미 크로스플랫폼이라 **맥에서 그대로 돈다**. 윈도우 종속은 ① `faster-whisper(CUDA)` ② `.ps1` 래퍼 3종뿐.
- **전사 엔진: `mlx-whisper` + `mlx-community/whisper-large-v3-turbo` 채택.** Apple MLX가 M4 Max의 **Metal GPU**를 직접 쓴다. `pip` 한 줄 설치, 네이티브 파이썬 API, `whisper.cpp` 대비 약 2배. → **`faster-whisper`는 맥에서 폐기**(CTranslate2에 Metal/MPS 백엔드가 없어 CPU-only. 이게 이식이 안 되는 근본 이유).
- **`.ps1` 오케스트레이터 3종(`yt_fetch`·`transcribe`·`run_youtube`)은 파이썬 단일 오케스트레이터로 재작성**해 PowerShell 의존을 제거한다. **산출물 계약(artifact contract)은 100% 동일**하게 유지 → 다운스트림(노트 작성·`rebuild_index.py`)은 손 안 댐.
- **더 이상 시그마(윈도우 GPU)로 오프로드할 필요가 없다.** 맥이 메인 브레인이므로 전사까지 로컬에서 끝난다.
- **고도화 방향**: 한글자막/영어 자막 우선 활용(전사 스킵 가능성 탐침) · 페치↔전사 파이프라이닝 · macOS 26 `SpeechAnalyzer` 네이티브 옵션 · 전사 후 학습노트 자동 초안 · 야간 스케줄(launchd).

### 2.1 후보 비교 (웹·GitHub 리서치 2025–2026)

| 엔진 | 설치 | M-시리즈 속도 | 정확도 | 자동화 | 판정 |
|---|---|---|---|---|---|
| **mlx-whisper** (Apple MLX) | `pip install -U mlx-whisper` | **whisper.cpp의 ~2×** (turbo, Metal GPU); turbo 기준 10–20× 실시간 | large-v3=최상, turbo=근접 | ✅ 네이티브 py API+CLI, HF 자동 다운로드 | **채택** |
| whisper.cpp (Metal+CoreML) | `brew install whisper-cpp` | large-v3 2–3× 실시간, CoreML(ANE) 시 >3× | 동일 가중치=높음 | ◯ C++ 바이너리(shell out) | 대안①(무의존 바이너리) |
| Apple **SpeechAnalyzer** | 내장(macOS 26) + Swift CLI | MacWhisper turbo의 **~2.2×**, ANE | large-v3-turbo급 | △ Swift/CLI만(py 아님) | 대안②(무설치·저전력) |
| lightning-whisper-mlx | `pip install lightning-whisper-mlx` | 배치+양자화로 더 빠르다 주장 | 증류/양자화로 일부 손실 | ◯ py API | 보류(유지보수 약함) |
| insanely-fast-whisper (MPS) | `pipx install ...` | MPS 백엔드 불안정(`NotImplementedError`), CPU 폴백 | 높음 | ✗ 맥에서 불안정 | **제외** |
| faster-whisper (CTranslate2) | `pip install faster-whisper` | **맥은 CPU-only** (Metal 없음) ~3× 실시간 | 최상(현행과 동일) | ✅ | **폐기**(속도 손실·GPU 못 씀) |

### 2.2 채택: `mlx-whisper` + `large-v3-turbo`

근거(우리 케이스 = M4 Max, 영어 15–30분 강연 배치, 속도/정확도 균형):
- **가장 빠른 "파이썬 자동화가 쉬운" 옵션.** Metal GPU에서 whisper.cpp의 2배(벤치 검증).
- **`large-v3-turbo`가 스위트스팟** — 또렷한 영어 강연에서 large-v3에 근접, 디코더 레이어가 적어 훨씬 빠름.
- 순수 `pip`, 빌드/CUDA/CoreML 변환 불필요 → 배치 스크립트에 바로 삽입.
- **정확도가 필요한 어려운 강연은 모델 ID만 `mlx-community/whisper-large-v3`로 교체**(코드 동일, 문자열 1개).

### 2.3 왜 faster-whisper는 안 되나 (핵심)

`faster-whisper`는 CTranslate2 위에서 도는데, **CTranslate2에는 Metal/MPS 백엔드가 없다.** 맥에서 `device="mps"`는 `unsupported device mps` 에러, `device="cuda"`는 당연히 불가. 결국 CPU-only로만 돌아 M4 Max의 GPU/ANE를 한 톨도 못 쓴다. → 이식이 "안 되는" 게 아니라 "이식해도 느린" 케이스라 엔진 자체를 바꾼다.

## 3. 아키텍처 설계

### 3.1 원칙 — "산출물 계약 동일, 엔진만 교체"

새 전사 코어는 기존 `transcribe.py`와 **완전히 같은 산출물**을 같은 위치에 쓴다:

| 산출물 | 내용 | 다운스트림 소비자 |
|---|---|---|
| `transcript.txt` | 세그먼트 텍스트 개행 결합 | 노트 작성(Claude) |
| `segments.json` | `{model,device,duration,segments[{id,start,end,text}]}` | 노트·검증 |
| `transcript.srt` | 타임코드 자막 | 참고 |
| `transcribe.done` | 센티넬(`segments=.. duration=.. model=.. device=..`) | 멱등성 검사 |
| stdout `TRANSCRIBE_OK segments=N` | 성공 마커 | 오케스트레이터 |

→ 이 계약만 지키면 ⑤ 노트화, ⑥ 인덱스는 **무수정**으로 재사용된다. 이게 이식의 안전벨트.

### 3.4 전사 코어 `transcribe_mlx.py` — 골격

```python
"""Transcribe audio with mlx-whisper (Apple MLX / Metal GPU).
기존 _template/transcribe.py와 동일한 산출물 계약을 지킨다."""
import argparse, json, time
from pathlib import Path
import mlx_whisper

def fmt_ts(s):  # SRT 타임코드
    h=int(s//3600); m=int((s%3600)//60); x=s-h*3600-m*60
    return f"{h:02d}:{m:02d}:{x:06.3f}".replace('.', ',')

def main():
    ap = argparse.ArgumentParser()
    ap.add_argument("--workspace", type=Path, required=True)
    ap.add_argument("--language", default="en")   # "auto"→None
    ap.add_argument("--model", default="mlx-community/whisper-large-v3-turbo")
    a = ap.parse_args()
    ws = a.workspace.resolve()
    audio = ws / "audio.wav"
    lang = None if a.language == "auto" else a.language

    t0 = time.time()
    r = mlx_whisper.transcribe(
        str(audio),
        path_or_hf_repo=a.model,
        language=lang,
        word_timestamps=True,           # 단어 타임스탬프(노트 인용 정밀화)
        condition_on_previous_text=False,  # 긴 강연 루프 방지(현행과 동일 철학)
    )
    segs = [{"id": i+1, "start": s["start"], "end": s["end"], "text": s["text"].strip()}
            for i, s in enumerate(r["segments"])]

    (ws/"transcript.txt").write_text("\n".join(s["text"] for s in segs)+"\n", encoding="utf-8")
    (ws/"segments.json").write_text(json.dumps(
        {"model": a.model, "device": "mlx", "duration": r.get("duration"), "segments": segs},
        ensure_ascii=False, indent=2), encoding="utf-8")
    with (ws/"transcript.srt").open("w", encoding="utf-8") as f:
        for s in segs:
            f.write(f"{s['id']}\n{fmt_ts(s['start'])} --> {fmt_ts(s['end'])}\n{s['text']}\n\n")
    (ws/"transcribe.done").write_text(
        f"segments={len(segs)} model={a.model} device=mlx\n", encoding="utf-8")
    print(f"TRANSCRIBE_OK segments={len(segs)}  ({time.time()-t0:.1f}s)")

if __name__ == "__main__":
    raise SystemExit(main())
```

> 현행 `transcribe.py`의 VAD(`vad_filter`)는 mlx-whisper 기본 파이프라인엔 없다. 강연 음성은 대체로 문제없으나, 무음/박수 구간이 길면 옵션으로 silero-VAD 전처리를 붙일 수 있다(열린결정 §6).

### 3.5 페치·오케스트레이터 이식

- **`yt_fetch.py`** = `yt_fetch.ps1`을 1:1 파이썬화: `yt-dlp --js-runtimes node -x --audio-format wav --postprocessor-args "-ar 16000 -ac 1" --write-info-json` → `workspaces/yt-<id>/audio.wav`+`audio.info.json` → `extract_provenance.py` 호출 → `youtube.json`. (`run_ytdlp()` 헬퍼 재사용.)
- **`run_youtube.py`** = `run_youtube.ps1` 이식: 큐 JSON 로드 → 영상별 `yt_fetch.py` → `transcribe_mlx.py` 직렬(또는 파이프라인) → **산출물 검증으로 성공 판정**(현행 `.ps1` 검증 로직을 파이썬으로 흡수) → 멱등(`transcript.txt` 있으면 스킵) → 요약 출력.

## 5. 고도화 아이디어

1. **자막 우선 활용(전사 스킵 탐침)** — 채널이 `[한글자막]`을 붙인다. `yt-dlp --list-subs`로 **다운로드 가능한 ko/en 자막 트랙이 있으면** Whisper를 건너뛰거나 교차검증에 쓴다(무료·정확). 번인(하드섭)이면 불가 → 영상별 탐침 스텝. 성공 시 GPU 시간 0.
2. **페치↔전사 파이프라이닝** — 전사(GPU)가 도는 동안 다음 영상 오디오를 미리 다운로드(네트워크). 생산자-소비자 큐로 벽시계 단축. mlx는 단일 GPU라 전사 자체 병렬은 이득 적음.
3. **macOS 26 `SpeechAnalyzer` 트랙** — 무설치·저전력·ANE. Swift CLI(`argmaxinc/apple-speechanalyzer-cli-example`) 래핑해 `--engine speechanalyzer` 옵션으로 선택 가능하게. 대량 배치/저전력 야간 실행에 후보.
4. **전사 후 학습노트 자동 초안** — 현재 ⑤는 수동(Claude). `note_template.md`+`youtube.json`+`transcript.txt`를 소넷 서브에이전트에 넘겨 초안 자동 생성 → 차드는 검수만. (`note_workflow.js` 이미 존재 → 재배선.)
5. **야간 스케줄(launchd)** — `rss_watch → 자동 큐 → run_youtube → 알림`을 맥 launchd로. 시그마 윈도우 스케줄 의존 제거(맥이 메인 브레인).
6. **단어 타임스탬프 → 정밀 인용** — `word_timestamps=True` 활용해 노트의 "화자의 디테일" 섹션에 정확한 타임코드 인용.
7. **화자 다이어라이제이션(선택)** — 대담/인터뷰(예: Boris×Niklas)엔 화자 구분이 유용. 맥은 pyannote가 MPS 부분지원(대체로 CPU)이라 거칠다 → 단일화자 강연엔 불필요, 인터뷰만 옵션.
